🤖 Agent vs ⚡ Skill vs 🔧 Tool

Utensilios, recetas y cocineros. La analogía perfecta para entender los tres niveles de autonomía en IA.

🧱 La analogía de la cocina

🔧

Un utensilio de cocina. Hace UNA cosa.
Atomico Predecible Barato

🌳 Arbol de decision

❓ ¿La tarea requiere un solo paso o varios?
👉 Un solo paso
✅ Tool — Usa una funcion atomica: get_weather, send_email, query_db
👉 Varios pasos
❓ ¿Los pasos son siempre los mismos?
👉 Si, siempre iguales
✅ Skill — Cadena fija, predecible y mas barata que un agente
👉 Pueden variar o surgir imprevistos
❓ ¿Pueden surgir imprevistos o ramificaciones?
👉 Si, necesita adaptarse
✅ Agent — Se adapta, pero pon guardrails para controlarlo
👉 No estoy seguro
💡 Empieza con Skill y escala a Agent si se queda corto

🎯 ¿Que usarias tu?

Pon a prueba tu intuicion. Elige la opcion correcta para cada escenario.

Escenario 1: Necesitas que tu app envie un email de confirmacion cuando el usuario se registra.
Es una funcion atomica: una llamada, un resultado. No necesita multiples pasos ni decisiones.
Escenario 2: Quieres procesar facturas: extraer datos con OCR, clasificar por tipo, y guardar en la base de datos. Siempre los mismos 3 pasos.
Es una cadena fija de pasos que nunca cambia: OCR → clasificar → guardar. Perfecto para un Skill.
Escenario 3: Un usuario pide: "Organiza mi viaje a Tokio". El sistema debe buscar vuelos, comparar hoteles, verificar disponibilidad, ajustar si hay cambios de precio, y reservar.
Requiere decisiones dinamicas: si un vuelo no esta disponible, busca otro. Si el precio cambia, recalcula. Un Agent se adapta a cada situacion.
Escenario 4: Tu pipeline de CI necesita ejecutar tests, generar un reporte, y notificar por Slack. Los pasos nunca cambian.
Son multiples pasos fijos (test → reporte → notificar). Siempre iguales, sin variacion. Skill ideal.

🏢 Ejemplo: gestion de tickets de soporte

Una sola llamada. Para catalogar.

classify_ticket(text)
→
categoria + urgencia

Recibe el texto del ticket, devuelve su categoria y nivel de urgencia. Nada mas. Una funcion, un resultado.

Pipeline de 3 pasos fijos. Para el 80% de tickets estandar.

read_ticket
→
classify
→
draft_reply

Lee el ticket, lo clasifica, y genera un borrador de respuesta. Siempre los mismos 3 pasos, en el mismo orden. Predecible y eficiente.

Resolucion adaptativa. Para tickets complejos.

Leer ticket
→
Buscar FAQ
→
¿Encontro solucion?
No →
→
Buscar en JIRA
→
¿Es critico?
→
Escalar / Resolver

El agente decide en cada paso: si no encuentra respuesta en la FAQ, busca en JIRA. Si el ticket es critico, escala a un humano. Si tiene solucion, la aplica. No sigue un guion fijo.

Heuristica: si puedes escribir un script determinista para la tarea, no necesitas un agente. Empieza siempre por la opcion mas simple que funcione.
