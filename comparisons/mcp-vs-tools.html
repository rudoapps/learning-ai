🔌 MCP vs 🔧 Tools custom

¿Estandar abierto o implementación a medida? Depende de tu contexto.

🔌 La analogía USB

Antes de USB, cada periférico tenía su conector: impresoras con puerto paralelo, ratones con PS/2, cámaras con FireWire. Cada fabricante hacía su driver. MCP hace lo mismo para IA: un protocolo estándar para conectar cualquier herramienta a cualquier modelo.

🔧 Tools custom 🔌 MCP
Implementación Escribes la función directamente en tu backend Usas un servidor MCP que expone tools via JSON-RPC
Portabilidad Atada a tu stack y a un proveedor de LLM Funciona con cualquier cliente MCP (Claude, GPT, Gemini...)
Ecosistema Solo lo que tú escribes Cientos de servidores MCP open-source (postgres, github, slack, filesystem...)
Setup Más simple para 1-2 tools Requiere arrancar servidores MCP y configurar transporte
Mantenimiento Tú mantienes todo La comunidad mantiene los servidores populares
Control Total — es tu código Delegas la ejecución al servidor MCP
Latencia Directa (in-process o HTTP local) Un hop más (stdio/HTTP al servidor MCP)
Seguridad Tú controlas permisos y sandbox El servidor MCP gestiona sus propios permisos

🌳 Árbol de decisión

❓ ¿Cuántas integraciones necesitas?
1-2 integraciones simples → Tools custom (más rápido de montar)
❓ ¿Existe un servidor MCP oficial para lo que necesitas?
¿Tu integración es muy específica/propietaria? → Tools custom (envuelve tu lógica directamente)
Sí (postgres, github, slack...) → MCP (ya está hecho, solo configuras)
❓ ¿Cambiarás de proveedor de LLM en el futuro?
Probablemente → MCP (no reescribes las integraciones al cambiar de modelo)
No, estoy atado a uno → Tools custom (más simple, menos abstracción)

🧩 ¿Qué usarías tú?

1. Estás haciendo un bot interno que solo necesita consultar tu API REST propia para responder preguntas del equipo.

Una sola integración simple y propietaria — Tools custom es más directo y rápido de implementar.

2. Tu empresa usa Postgres, GitHub y Slack, y quieres que el asistente IA acceda a los tres.

Ya existen servidores MCP para Postgres, GitHub y Slack. Solo configuras, no programas.

3. Estás construyendo una plataforma SaaS donde los clientes pueden elegir entre Claude, GPT o Gemini, y todos deben tener las mismas herramientas.

MCP da portabilidad entre modelos: defines las tools una vez, funcionan con cualquier LLM compatible.

4. Necesitas una integración muy específica con tu sistema de facturación legacy que tiene una API SOAP compleja.

Demasiado específico para un servidor MCP genérico — Tools custom te da control total sobre la integración SOAP.

Plataforma multi-modelo

El usuario elige Claude o GPT, las tools deben funcionar igual

MCP ▶

Las tools se definen una vez y funcionan con cualquier LLM compatible con MCP. Sin reescribir adaptadores para cada proveedor. Portabilidad real.

Tendencia: MCP es joven (2024-2025) pero creciendo rápido. Si empiezas un proyecto nuevo con múltiples integraciones, MCP es la apuesta segura. Para una tool puntual, custom sigue siendo más pragmático.
